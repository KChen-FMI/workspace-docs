# LLMウィザード POC計画

## 目的
既存データ（社内規程文書とJIRAチケット、OneNote）を使って、LLMウィザードによるサービスカタログ更新の有効性を検証する。

## スコープ
- データソース:
  - 社内規程
  - JIRAチケット
  - OneNote
- 以下の評価基準でLLMウィザードを評価する。

スコープ外:
- 上記3件以外の新規データソース追加
- 本番環境への完全統合
- POC範囲を超えた自動化

## 評価基準
1. **情報抽出**
   - 規程・JIRAチケット・OneNoteから必要な情報を安定して抽出できるか
2. **抽出情報のマッチング**
   - 抽出情報を正しい既存サービスに紐づけられるか
3. **スコープ判定**
   - カタログ対象かどうか、どのカテゴリに属するかを判断できるか
4. **アクティビティ定義**
   - サービスに対する妥当なアクティビティを提案できるか

## 入力
- 規程サンプル（PDFまたはテキスト）
- JIRAチケットサンプル（エクスポートまたはAPI）
- OneNoteサンプル（エクスポートまたはAPI）

## 出力
- アイテムごとの抽出JSON
- サービス紐づけ提案（信頼度付き）
- スコープ/カテゴリ提案
- アクティビティ提案リスト

## POC手順（案）
1. 規程サンプル10〜20件、JIRAチケットサンプル20〜50件、OneNoteサンプル5〜10件を用意する
2. サンプルに対してLLMウィザードを実行し、JSON出力を取得する
3. 期待結果と照合して手動レビューする
4. 評価基準ごとにスコアを付ける
5. 結果をまとめ、次ステップを判断する

## 成功基準
- サンプルに対する抽出精度が80%以上
- サービス紐づけの正解率が70%以上
- 70%以上の項目で明確なアクティビティ提案が得られる

## リスク
- ソースデータ品質のばらつき
- サービス定義のあいまいさ

## 次アクション
1. サンプルデータ件数の確定
2. 評価基準の採点ルール策定
3. レビュー会の設定
