# LLM抽出 PoC（最小構成）

## 目的
- LLM抽出の精度と運用性を、少数文書で検証する
- ルールベースとの違いを明確化し、運用判断に必要な根拠を作る

## 対象文書（3〜5件）
以下の5件を推奨（条文型＋方針型の混在で差が出やすい）:
- `01-0080_内部統制システムの基本方針_20240327.pdf`
- `03-0130_在宅勤務規程_20251001改正.pdf`
- `03-0131_遠隔地勤務規程_20251001改正.pdf`
- `07-0090_情報セキュリティ基本方針_20130207.pdf`
- `07-0120_情報システム管理規程_20250301.pdf`

## 実施ステップ
1. **LLM抽出**
   - Azure OpenAI APIで本文→JSON（`schema.md`準拠）
2. **人間レビュー**
   - 抽出結果をレビューし、誤り/漏れ/解釈違いを修正
3. **評価**
   - 重要フィールドの精度と修正負荷を測定
4. **結論**
   - LLM抽出の有効性・運用可否を報告

## 評価指標（最小）
- **必須フィールドの充足率**
  - `title`, `doc_id`, `doc_type`
- **主要フィールドの網羅性**
  - 責務（`responsibilities`）
  - 禁止事項（`prohibitions`）
  - 報告トリガー（`reporting_triggers`）
  - 必須行動（`required_actions`）
- **修正コスト**
  - 人手修正の行数/件数
- **所要時間**
  - 抽出→レビュー完了までの時間

## 成果物
- LLM抽出JSON（`*.llm.extracted.json`）
- 人間レビュー後の確定JSON（`*.reviewed.json`）
- 評価サマリー（表）
- レビューテンプレート（`run-book/POC/llm-extraction-review-template.md`）

## 判定基準（例）
- 必須フィールドの欠落が0件
- 主要フィールドの修正が「軽微」で済む
- 文書あたりのレビュー時間が許容範囲

## レビュー備考
- `reviewer_notes` に担当者の意見・判断を記入する

## 次のアクション候補
- PoC合格: スキル化（LLM抽出スキル作成）
- PoC不合格: ルール/プロンプト改良またはハイブリッド化
